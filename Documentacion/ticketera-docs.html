Resumen del sistema

Ticketera es una plataforma interna de gestión de tickets de soporte desarrollada para Equilybrio. Permite registrar, asignar y resolver solicitudes de soporte organizadas en bandejas temáticas, con seguimiento de SLA, base de conocimiento integrada y reportes operacionales.

Usuarios

3 roles: Admin, Agente y Cliente. Cada rol tiene permisos granulares configurables.

Bandejas

Tickets organizados en 7 bandejas temáticas: IT, Dev, DevOps, Analytics, Finanzas, RRHH y Seguridad.

SLA

Reglas de SLA por prioridad y área. El deadline se calcula automáticamente al crear cada ticket y se puede pausar.

Conocimiento

Base de artículos con categorías, búsqueda full-text y ABM completo para administradores.

Reportes

Dashboard en tiempo real con KPIs, distribución por bandeja, prioridad y rendimiento por agente.

Logging

Cada request HTTP y acción de negocio queda registrada en stdout con timestamp, color por status y usuario.

Autenticación

El sistema usa JWT (JSON Web Tokens) con expiración configurable (por defecto 8h). El token se almacena en sessionStorage bajo la clave ticketera_session.

Flujo de login

1. Cliente → POST /api/auth/login  { email, password }
2. Backend verifica password con bcryptjs
3. Backend responde { token: "JWT...", user: { id, name, email, role, role_label } }
4. Frontend guarda en sessionStorage
5. Cada request incluye:  Authorization: Bearer <token>

Motor SLA

Cálculo automático al crear

Al crear un ticket, el backend busca la regla SLA activa más específica para esa combinación de priority + dept. Las reglas de departamento específico tienen prioridad sobre las reglas "all".

// Lógica de selección de regla
const rules = await prisma.slaRule.findMany({ where: { priority, active: true } });
const rule  = rules.find(r => dept && r.dept === dept) || rules.find(r => r.dept === 'all');
const mins  = parseSlaMinutes(rule?.res);  // "4h" → 240, "30m" → 30
if (mins) deadline = new Date(Date.now() + mins * 60000).toISOString();

Pausa y reanudación

Al cambiar el estado a paused se guarda el timestamp en sla_paused_at. Al reanudar (cualquier otro estado), el backend calcula la duración de la pausa y extiende sla_deadline en ese mismo tiempo. El reloj continúa exactamente donde se detuvo.

// Al pausar
ticketData.sla_paused_at = new Date();

// Al reanudar
const pausedMs    = Date.now() - new Date(existing.sla_paused_at).getTime();
const newDeadline = new Date(new Date(existing.sla_deadline).getTime() + pausedMs);
ticketData.sla_deadline  = newDeadline.toISOString();
ticketData.sla_paused_at = null;
